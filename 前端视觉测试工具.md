# 前端视觉测试工具：TestSprite、Chromatic、Percy

> 演讲中提到的三个工具的独立说明文档，供学习参考。

---

## 一、它们各自在解决什么问题？

前端测试有一类特殊的难题：**代码功能对了，但页面看起来不对**。间距偏了、某个浏览器下布局塌了、动画缓动不对——这些靠单元测试根本抓不到，只能靠人眼。

这三个工具都是在解决这个问题，但角度不同：

| 工具 | 核心定位 | 依赖前提 |
|------|----------|----------|
| **Chromatic** | 基于 Storybook 的视觉回归测试 | 必须有 Storybook |
| **Percy** | 通用视觉回归测试，接入已有测试框架 | Playwright / Selenium / Cypress 等 |
| **TestSprite** | AI 自主生成并运行端到端测试 | 无特殊前提，接入 Cursor/VSCode |

---

## 二、Chromatic

**一句话**：给 Storybook 组件做视觉快照，每次提交自动对比，有差异就报警。

### 工作方式

Storybook 里每个 Story 就是一个组件状态的截图用例。Chromatic 在每次 push 时把这些 Story 全部渲染截图，和上一次的基线对比，找出像素级的变化。你在 UI 上审查差异，决定"这是有意的改动"还是"这是 bug"。

### 核心能力

- **多浏览器**：Chrome、Firefox、Safari、Edge 并行跑，一次发现跨浏览器差异
- **多视口**：可配置断点，自动测响应式
- **TurboSnap**：只对有代码变动的组件截图，不全量跑，速度快
- **PR 集成**：在 GitHub/GitLab PR 里直接展示视觉 diff，reviewer 一目了然

### 适合谁

- 项目里已有 Storybook 的团队
- 在维护设计系统、组件库的团队
- 需要设计师和开发者一起审查 UI 变动

### 局限

- **必须有 Storybook**，没有就用不了
- 基线需要人工确认，不能自动判断"这个变化对不对"
- 布局变动（如组件整体位移）误报率较高

### 免费额度

5,000 张截图 / 月，对小团队够用。

---

## 三、Percy

**一句话**：在你已有的 E2E 测试里插一行代码，Percy 负责截图、存储、对比和审查。

### 工作方式

Percy 不替代你的测试框架，而是嵌入进去。你用 Playwright 或 Selenium 写好测试流程，在关键页面调用 `percySnapshot(page, '页面名')` ，Percy 把这些截图上传到云端，和上次基线对比，生成可视化的 diff 报告。

```bash
# 安装（以 Playwright 为例）
npm install -D @percy/cli @percy/playwright

# 运行
npx percy exec -- npx playwright test
```

### 核心能力

- **真实设备**：依托 BrowserStack 的 50,000+ 真实浏览器和设备，不是模拟
- **Visual AI Engine**：AI 过滤动态内容（轮播图、广告、时间戳）的噪音，减少误报
- **Visual Review Agent**：AI 辅助审查，给出自然语言描述"这里的变化是由于 CSS margin 改了"，评审速度提升 3 倍
- **跨框架**：Playwright、Selenium、Cypress、Storybook 都支持

### 适合谁

- 已有 E2E 测试，想加视觉回归层
- 需要覆盖真实设备（移动端响应式）
- 大团队，需要统一的 UI 审查工作流

### 局限

- 免费额度用完后价格较高（付费从 $199/月起）
- 不自动生成测试用例，需要你自己写 E2E 测试

### 免费额度

5,000 张截图 / 月。

## 四、TestSprite

**一句话**：AI 直接帮你生成测试用例、在云端跑、发现问题告诉你怎么修——你不用写一行测试代码。

### 工作方式

TestSprite 的定位是"AI 测 AI 写的代码"。你在 Cursor 或 VSCode 里通过 MCP 调用它，给它一句话指令（比如"帮我测这个登录流程"），它会：

1. 读你的代码或 PRD，理解功能意图
2. 自动生成测试计划和测试脚本
3. 在云端沙箱里执行测试
4. 把失败的用例、截图、日志和修复建议返回给你

### 核心能力

- **意图理解**：不是像素比对，而是理解"用户点了这里，应该跳转到哪"
- **全栈覆盖**：前端 UI、后端 API、数据层都能测
- **自动修复脆弱选择器**：测试脚本因 DOM 变化失效了，自动更新选择器
- **CI/CD 集成**：可以跑在流水线里，持续验证
- **MCP 集成**：在 Cursor 里直接调用，不用切工具

### 适合谁

- 大量使用 AI 生成代码，需要验证层的团队
- 没有专职 QA、想自动化 QA 流程的小团队
- 不想手写测试脚本的开发者

### 局限

- 工具还在早期（2026 年），复杂领域逻辑偶尔误判
- 本地项目需要额外的隧道工具才能测
- 按 credit 计费，大型 CI/CD 流水线消耗快
- TypeScript / Python 支持最好，其他语言还在完善

### 定价

免费版 150 credits；付费从 $29/月（个人）起。

## 五、三者横向对比

| | Chromatic | Percy | TestSprite |
|---|---|---|---|
| **测什么** | 组件视觉状态 | 页面视觉状态 | 功能 + 视觉 + 交互 |
| **谁生成测试** | 你写 Story | 你写 E2E | AI 自动生成 |
| **对比方式** | 像素 diff | 像素 diff + AI 降噪 | 意图理解 + 视觉 |
| **前提依赖** | 需要 Storybook | 需要 E2E 框架 | 无 |
| **接入难度** | 低（已有 Storybook） | 中（改造现有测试） | 低（一句话指令） |
| **免费额度** | 5,000 截图/月 | 5,000 截图/月 | 150 credits |
| **成熟度** | 高 | 高 | 早期 |

---

## 六、实际怎么选？

**如果你们已经在用 Storybook** → 先试 Chromatic，门槛最低，价值最直接。

**如果你们有 Playwright/Cypress E2E 测试** → 加 Percy，补视觉回归层，不需要改测试架构。

**如果你们大量用 Cursor 生成代码，但几乎没有测试** → TestSprite 是最快能建立验证层的方式，但要接受它还在早期的风险。

**三者不互斥**，成熟团队可以组合使用：Chromatic 管组件，Percy 管页面流程，TestSprite 管 AI 生成代码的验证。
