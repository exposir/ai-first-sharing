# 内部 QA Agent 产品 + 技术思路

> 初步整理，后续细化

---

## 一、背景和问题

AI 辅助开发让代码产出速度大幅提升，但验证速度没有跟上。现有的商业工具（TestSprite 等）是通用的，每次只能拿到片段上下文，无法理解产品的历史决策和设计原则。

前端验证尤其困难：视觉正确性和交互正确性无法被简单形式化，纯靠人眼审查是当前最大瓶颈。

---

## 二、核心思路

**让 Agent 贯穿整个开发流程，而不是只在测试环节介入。**

```
产品文档
   ↓ Agent 解析意图、提取功能点
需求（结构化）
   ↓ Agent 细化边界条件
技术方案
   ↓ Agent 参与设计决策
代码（Cursor 生成）
   ↓
QA Agent 测试
   ↓ 基于完整上下文生成用例、执行、返回结果
```

Agent 全程在场，到测试环节时已经知道：
- 功能最初的意图是什么
- 需求里讨论过哪些边界条件
- 技术方案里做了哪些取舍

所以生成的测试用例不是从代码反推，而是从意图推导，准确度更高。

---

## 三、和商业工具的差异

| | TestSprite（线上版） | 内部 QA Agent（本地版） |
|---|---|---|
| 运行环境 | 云端沙箱 | 本地 / 内网 |
| 上下文来源 | 当前代码 + 你给的描述 | 产品文档 + 需求历史 + 技术决策 + 代码 |
| 适用范围 | 通用，任何项目 | 专用，深度适配内部产品 |
| 访问本地服务 | 需要 ngrok 等隧道 | 直接访问 localhost |
| 数据安全 | 代码上传到第三方 | 全部在内部 |
| 核心优势 | 开箱即用 | 上下文更深，理解更准 |

---

## 四、最小可行版本（技术栈）

**核心链路：**

```
触发（Cursor MCP 或脚本）
   → LLM 根据上下文生成 Playwright 测试脚本
   → 本地执行，截图 + 断言
   → 结果返回 IDE（失败原因 + 截图 diff）
```

**技术选型：**

- **执行层**：Playwright（跑测试 + 截图）
- **生成层**：LLM API（Claude / GPT，根据上下文生成测试脚本）
- **接入层**：MCP Server（接入 Cursor，一句话触发）
- **基线管理**：本地文件存截图，简单 diff 对比
- **上下文存储**：产品文档、需求文档作为 Agent 的背景知识

**最小版本不需要：**
- 云端部署
- 精美的 diff UI
- 多用户权限
- 计费系统

---

## 五、核心挑战

1. **测试脚本生成质量** — LLM 初期误报多，需要 prompt 调优和规则过滤
2. **上下文管理** — 产品文档和历史决策如何结构化存储、如何有效检索
3. **基线维护** — 有意的 UI 改动如何更新基线，避免每次都要手动确认

---

## 六、价值主张

> 不是技术更强，而是上下文更深。

商业工具每次从零理解你的代码；内部 Agent 知道产品的设计原则、历史遗留、边界决策——这才是真正的质量保障，不靠工具有多聪明，靠的是信息的完整性。
